# Treiber Stack - 并发栈

## 模块功能

Treiber Stack 是一个基于 CAS（Compare-And-Swap）重试循环实现的并发栈，提供线程安全的压入（push）和弹出（pop）操作。该实现在 CPython 环境下采用细粒度锁保护 CAS 核心步骤，锁外完成所有预构造工作，避免将整个 push/pop 操作串行化，适用于高并发场景下需要高效栈操作的场合。

### 核心特性

- **细粒度锁设计**：锁仅保护 CAS 的比较-赋值核心步骤，所有对象预构造在锁外完成，临界区极小
- **CAS 重试循环**：外层采用经典 Treiber 自旋重试循环，CAS 失败立即重新读取状态并重试
- **ABA 防护**：使用带版本号的不可变引用防止 ABA 问题
- **并发安全**：多线程并发操作保证数据一致性
- **内存安全**：与 Python GC 兼容，无需手动内存管理
- **泛型支持**：支持存储任意类型的元素

## 核心类职责

### `TreiberStack[T]`

线程安全的并发栈实现，支持泛型类型 `T`。

#### 主要方法

| 方法 | 描述 |
|------|------|
| `push(value: T) -> None` | 将元素压入栈顶 |
| `pop() -> Optional[T]` | 弹出并返回栈顶元素，栈空时返回 None |
| `peek() -> Optional[T]` | 返回栈顶元素但不移除，栈空时返回 None |
| `is_empty() -> bool` | 判断栈是否为空 |
| `size() -> int` | 返回栈中元素数量（并发场景下为近似值） |

### 内部类

#### `_Node[T]`

单向链表节点，存储栈元素值和指向下一节点的引用。

#### `_TaggedReference[T]`

带版本号的不可变引用包装类，用于防止 ABA 问题。包含：
- `node`: 指向栈顶节点的引用
- `version`: 版本号，每次成功更新栈顶后递增

该类被标记为 `frozen=True`（不可变），每次栈顶更新都会创建全新的 `_TaggedReference` 实例。

## 算法原理

Treiber Stack 由 R. Kent Treiber 在 1986 年提出，是最简单和最经典的无锁数据结构之一。

### 核心思想

1. **链表结构**：使用单向链表存储栈元素，栈顶指针指向链表头部
2. **细粒度 CAS 原语**：CPython 缺少硬件级 CAS 指令，采用细粒度锁仅保护"比较身份 + 原子赋值 + size 更新"三个核心步骤，锁外完成所有对象预构造
3. **失败重试**：如果 CAS 返回失败（栈顶已被其他线程修改），则重新读取当前状态并重试

### Push 操作流程

```
1. 创建新节点，设置新节点的值
2. 循环：
   a. 读取当前栈顶引用对象（包含节点指针和版本号）
   b. 设置新节点的 next 指针指向当前栈顶节点
   c. 预构造新的 TaggedReference 实例（新版本号 + 新节点）
   d. 调用 CAS 原语：在细粒度锁内比较当前_head是否仍等于预期值
   e. 如果 CAS 成功，size 已同步更新，直接返回
   f. 如果 CAS 失败，继续循环重试
```

### Pop 操作流程

```
1. 循环：
   a. 读取当前栈顶引用对象（包含节点指针和版本号）
   b. 如果栈顶为空，返回 None
   c. 获取栈顶节点的 next 节点，暂存要返回的值
   d. 预构造新的 TaggedReference 实例（新版本号 + next 节点）
   e. 调用 CAS 原语：在细粒度锁内比较当前_head是否仍等于预期值
   f. 如果 CAS 成功，size 已同步更新，返回暂存的值
   g. 如果 CAS 失败，继续循环重试
```

## CAS 实现机制与重试策略

### CAS 实现机制

由于 CPython 语言层面缺少真正的硬件级 Compare-And-Swap 指令，本实现采用以下折中方案模拟 CAS 语义：

1. **锁外预构造**：新的 `_TaggedReference` 实例、新节点等所有对象都在获取锁之前构造完成，临界区内不做任何内存分配
2. **细粒度临界区**：`_cas_lock` 仅保护"身份比较 + 引用赋值 + size 更新"三步，临界区长度极小
3. **身份比较**：使用 `is` 运算符比较栈顶引用的对象身份。由于 `_TaggedReference` 是不可变（frozen）类，每次修改必然创建新实例，因此对象身份相同等价于状态未被修改
4. **size 同步更新**：将 `_size` 的增减也放在 CAS 临界区内，确保每次 CAS 成功时 size 也被原子更新，避免单独的计数器竞争

这种实现与"用全局锁串行化整个 push/pop"有本质区别：全局锁会将节点构造、链表链接等所有操作都包在锁内，而本实现的锁仅保护几条基本赋值语句，锁持有时间缩短了几个数量级。

### 与真正硬件 CAS 的对应关系

| 硬件 CAS (CMPXCHG) | 本实现对应 |
|-------------------|-----------|
| 比较内存值与期望值 | `self._head is expected` 身份比较 |
| 条件满足写入新值 | `self._head = new_ref` 引用赋值 |
| 整个操作原子性 | 由 `threading.Lock` 保证比较与赋值的原子性 |

### 重试策略

采用**无限制自旋重试**策略：

1. 每次操作在 `while True` 循环中执行
2. 读取当前栈顶引用对象
3. 预构造新状态，调用 CAS 尝试提交
4. 成功则退出循环，失败则重新读取状态并重试

该策略的优势：
- 实现简单，正确性易于证明
- 在低到中等并发下性能优异
- 避免了复杂的退避算法和线程上下文切换

## ABA 问题的防护机制

### 什么是 ABA 问题

在无锁算法中，ABA 问题指的是：
1. 线程 A 读取位置 X 的值为 A
2. 线程 B 修改位置 X 的值为 B
3. 线程 B 又将位置 X 的值改回 A
4. 线程 A 执行 CAS，发现值仍然是 A，误以为没有变化而成功

这会导致潜在的数据不一致，因为虽然值相同，但中间状态已经发生了变化。

### 带版本号引用的防护方案

本实现采用**带版本号的不可变引用（Tagged Reference）**机制：

1. 将栈顶指针和一个单调递增的版本号打包在不可变对象中
2. 每次成功更新栈顶时，都创建**全新的** `_TaggedReference` 实例，版本号递增 1
3. CAS 比较时使用对象身份比较（`is`）：
   - 判定依据：当前 `self._head` 是否与线程读取时的 `expected` 是**同一个内存实例**
   - 版本号的作用：确保每次修改必然产生与之前不同的新实例
   - 即使两个 `_TaggedReference` 的 node 和 version 都相同，只要不是同一个对象实例，`is` 也会返回 False
4. 因为每次修改都产生新实例，而版本号保证即使回到相同 node 也会产生不同实例，身份比较必然失败，彻底杜绝了 ABA 问题

#### 身份比较与版本号的关系辨析

- **身份比较 (`is`) 是 CAS 的判定依据**：直接比较两个引用是否指向内存中同一个对象实例。这是判定"栈顶状态是否被其他线程修改过"的根本标准
- **版本号是确保产生不同实例的机制**：单调递增的版本号保证了每次成功更新都会构造出内容不同的 `_TaggedReference`，从而必然是不同的对象实例
