产物不满意：
1. pubsub.py 的 _enqueue_for_subscriber 在 DROP_OLDEST 分支弹出旧消息后却用新消息的 message_id 写入丢弃记录，实际复现中同一条新消息同时出现 dropped 和 success，真正被挤掉的旧消息没有准确状态，破坏逐消息逐订阅者投递追踪。根因是丢弃时忽略 popleft 返回的旧消息，应记录被移除消息的 id 并保证同一投递对象状态不互相矛盾。
2. pubsub.py 的 _enqueue_for_subscriber 在 BLOCK 策略下只执行 pass 后继续追加消息，缓冲区大小 1 时可增长到 4 完全绕过缓冲上限，慢订阅者无限堆积内存既无限流也没隔离意义。根因是暴露了公共策略枚举没实现对应行为，应使用条件变量等待可用空间或删除该策略并在文档中保持一致。
3. tests/pubsub/test_pubsub.py 的背压用例只断言缓冲区不超上限或存在 DROPPED 状态，没校验被丢弃消息身份也没校验同一消息不能同时成功和丢弃。因此 60 个测试通过仍漏掉投递记录错绑问题。根因是测试只覆盖数量现象未覆盖状态语义，应按消息 id 构造满缓冲场景并逐条断言最终投递记录。
过程不满意：
1. 模型写完 pubsub.py 后直接根据自己实现编写测试再用这些测试证明功能完成，背压测试没从提示词记录每条消息对每个订阅者投递状态反推精确断言，结果测试只检查列表长度和状态存在性掩盖 DROP_OLDEST 记录错绑。应先列出每种背压策略下每条消息期望状态再据此写测试。
2. 模型在设计阶段额外加入 BLOCK 策略并在 README 中说明该策略，但实现阶段没补充真实阻塞限流或丢弃逻辑，测试阶段也没覆盖该枚举路径，用户选择该策略就得到无上限缓冲。根因是扩展了提示词未要求的公共能力却缺少闭环验证，应收敛到已实现策略或为新增策略补齐实现文档和测试。
3. 模型在测试验证阶段大量使用 time.sleep 等待异步线程并在 60 个用例通过后立即宣称完成，没继续用确定性同步检查慢订阅者阻塞时缓冲和投递记录，线程调度差异掩盖真实边界问题也难以稳定复现背压错误。应使用 Event 或 Barrier 控制处理器停顿点再断言缓冲内容和记录状态。
