产物不满意：
1. rewriter.py 的 _extract_path 方法虽然引入了 urlparse 做结构化解析，避免了在 scheme、host 等组件上的误匹配，但返回值仍包含查询参数部分：`f"{path}?{parsed.query}"`。提示词第 3 点明确要求将匹配限定到路径部分以排除查询参数中的误匹配，当前实现并未做到。当 URL 为 `http://example.com/products?filter=api` 且改写规则为 `r"/api"` 时，_extract_path 返回 `/products?filter=api`，`self.pattern.search(path)` 仍会在 query string 中命中 "api" 而错误返回 True，同时 rewrite 方法中的 `rule.pattern.sub(rule.replacement, path_with_query)` 也会对查询参数执行替换。根因是模型只考虑了去掉 URL 中的协议和域名部分，未区分 path 和 query 的语义边界，应改为 `_extract_path` 只返回 `parsed.path`，matches 和 rewrite 中后续的匹配和替换也相应只作用于路径部分，查询参数保持原样透传。
