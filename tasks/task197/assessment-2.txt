产物不满意：

1. then 改为不展平后，README 的描述段仍写"回调返回 Future 时自动展平"，与代码实际行为矛盾。用户按文档理解以为 Future 返回值会被自动解包，实际拿到的是 Future 对象本身，后续值类型操作会报类型错误。根因是修改代码后只处理了 API 表的条目移除，忘记了同步更新 then 的行为描述段落。应将描述改为 Future 作为普通值传递不展平，并删除误导性的"与 compose 行为一致"注释。

2. catch 方法创建的 _ChainLink 未显式传 flatten，默认值 False 导致回调返回 Future 时不再展平。提示词只要求 then 不展平、compose 展平，没要求改 catch。原本 catch 返回 Future.resolve 后链式值是字符串，现在变 Future 对象，原有断言被迫从期待字符串改为期待 Future。根因是增加 flatten 判断时没在 catch 处显式传 flatten=True。应为 catch 添加 flatten=True。

3. README 的 then/compose/catch 对比表中，then 的回调返回到仍写"任意值 / Future"，未注明 Future 不会被展平这一核心差异。这个表格是用户选择方法的第一入口，当前信息不足以让用户快速区分 then 和 compose 的关键差异，需要读完大段文字才能理解。根因是修改代码语义后只更新了 API 表，对比表未被同步修订。应在 then 的对应列标注 Future 不展平，与 compose 形成对比。

过程不满意：

1. 模型修改传播方法增加 flatten 条件判断时，无意将 catch 的行为从展平改为不展平。提示词的修复要求只涉及 then 和 compose 的职责分离，从未提 catch 需要改变。模型发现副作用后没有回退，而是将原有测试断言从期待字符串改为期待 Future，属于用测试适配未授权的行为变更。根因是在修改传播逻辑时未区分范围边界，将展平判断笼统改为依赖 flatten 标志。应只对 then 的 _ChainLink 做定向修改，catch 保持原有行为不变。

2. 模型编辑 README 时只从 API 表删除了两个内部方法条目，未系统性审查 then 和 catch 的行为描述段落以及对比表是否仍然准确。直接后果是 then 描述说自动展平而代码不展平，catch 描述也说自动展平而代码也不展平，两处文档与实现不一致。根因是将三条要求当独立任务线性执行，改完代码后没有做关联修改的一致性检查。应在修改函数语义后全局审查 README 中所有受影响的内容。

3. 模型在第14步到第16步之间连续三次调用 view_files 读取同一个 README.md 文件，且这三次读取行为相同、目的相同，仅仅是每次读取后做了不同位置的修改。第一次读取已经获取了文档的完整内容，后续修改应基于已有的认知直接编辑，不需要反复回到文件读取相同内容。根因是模型缺乏工具调用间的记忆机制，每步操作前习惯性地重新读取文件确认内容。这种做法浪费了工具调用次数和上下文窗口，属于典型的低效推理模式。

4. 模型收到提示词后第一步直接用 todo_write 创建 4 项子任务，第二步才开始阅读源码。提示词要求修改三处相互关联的方法，应先读完完整代码和现有测试，理清调用链和影响范围后再规划。根因是把创建 TodoList 当作任务开始的仪式性动作，跳过了必不可少的前置分析环节。正确做法是先阅读相关源文件和测试代码，基于实际代码结构制定有针对性的修改计划。
