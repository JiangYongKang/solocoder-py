你实现的消息队列模块有几处需要修一下：
1. get_queue_size 方法返回的数量包含了 IN_FLIGHT 状态（已被消费者取走但未确认）的消息。这些消息当前不可被消费，计入队列大小会导致调用方对可用消息数产生误判。修复后该方法应只返回真正可以被 dequeue 消费的消息数量。
2. 死信队列中的消息 ID 与去重检查各自独立管理。当一条消息因重试超限进入死信队列、且其去重窗口过期后，同 ID 的新消息可以重新入队，但死信队列中仍残留旧版本的同 ID 消息，导致后续查询死信队列时出现同一 ID 的新旧消息混杂。需要清理死信队列中已有的同 ID 旧消息，或在去重逻辑中纳入死信队列的考量。
3. 清理包内未使用的死代码：QueueEmptyError 异常类定义了但从未被抛出，Message 上的 generate_receipt_handle 方法定义了但从未被任何消费方调用。这些代码增加了理解负担。
4. 修复后重新运行测试，需要让现有测试全部通过。如有必要，补充新的测试用例覆盖上述修复场景。
