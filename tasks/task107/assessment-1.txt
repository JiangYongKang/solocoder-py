产物不满意：
1. serializer.py 的 `_skip_unknown_field` 方法无法正确跳过未知的 STRING 或 BYTES 类型字段。当未知字段是长度大于等于 128 的字符串或字节数组时，其长度前缀的 varint 编码首字节会大于等于 0x80，方法进入第三步 `try: decode_uvarint(buf)` 成功解码了长度值后就直接 return 了，完全没有跳过后面的实际数据字节。这会导致后续所有字段的反序列化完全错位，读取到的是字符串内容当作 field_id，必然抛出 `DeserializationError` 或返回完全错误的数据。根因是序列化格式未在字段头部编码类型信息（类似 Protobuf 的 wire type），跳过逻辑仅靠首字节做启发式猜测，根本不可能可靠区分多字节 varint 整数字段和长字符串/字节数组字段。正确做法要么在序列化时为每个字段写入类型标记，要么在反序列化时让调用方传入 writer 的 Schema 以便通过 field_id 查询字段类型后精确跳过。现有测试 `test_unknown_fields_ignored` 只用 INT64 值 42 作为未知字段，恰好避开了这个问题，缺少未知 STRING 和 BYTES 字段的覆盖。
2. serializer.py 的 `deserialize` 方法（第 109-112 行）使用 `decode_uvarint(buf)` 读取了 `stored_version` 变量，但从第 114 行开始到方法结束，`stored_version` 再也没有被引用过。序列化格式中写入的 schema 版本号在反序列化时被完全忽略，成为一个无效字段。这意味着即使传给 `reader_schema` 参数声称数据的 schema 版本是 v1，但二进制数据实际是由 v3 schema 序列化产生的，代码也仅依赖外部的 `check_compatibility(writer_schema, reader_schema)` 调用来保证一致性，而不会用数据中自带的版本号做任何校验。根因是模型只完成了版本号的写入逻辑，但遗漏了版本号的读取和使用。应该在反序列化时用 `stored_version` 与 `reader_schema.version` 做比对，至少给出警告或在校验不一致时报错。
3. serializer.py 的 `_skip_unknown_field` 方法中异常重试逻辑（第 190-199 行）存在严重的缓冲区状态错误。方法先在第一个 try 块中调用 `decode_uvarint(buf)`，如果这个调用在读取了若干字节后抛出异常，buf 的读指针已经被向前移动了。紧接着第二个 try 块又调用 `length = decode_uvarint(buf)` 尝试重新读取，此时读指针已经移到了一个错误的位置，解码出来的 length 值毫无意义，再用 `buf.skip(length)` 跳过的字节数完全随机。根因是对有副作用的函数（修改 buf 读位置）进行捕获异常后重试，状态已经不可恢复。正确做法是在读取前先用 `peek_byte` 在不移动指针的情况下探测数据格式，探测失败则直接抛错，而非在已损坏的状态上继续尝试。
4. serializer.py 的 `deserialize` 方法第 124-126 行存在冗余逻辑：`field = self._fields_by_id.get(field_id)` 在 `reader_schema is not None` 时立即被 `field = reader_schema.field_by_id(field_id)` 无条件覆盖，前一行赋值成为死代码。当 `reader_schema` 传入后，字段匹配完全不再依赖 writer 的 schema，而 `self._fields_by_id` 是基于构造器中传入的 writer schema 构建的索引，这个索引在 reader_schema 被传入后完全被绕过了。虽然功能上不影响结果，但代码逻辑混乱，容易在后续维护中引入误解（例如误以为会做双 schema 字段的交集校验）。应简化为：只有当 `reader_schema is None` 时才使用 `self._fields_by_id`。
5. serializer.py 的 `_write_field_value` 方法（第 148-163 行）对所有字段值做了强制类型转换：`bool(value)`、`int(value)`、`str(value)`、`bytes(value)`。这种静默转换会掩盖调用方的数据错误，例如往 BOOL 字段传入字符串 "false" 会被转换为 True（非空字符串为真值），往 INT64 字段传入浮点数 3.14 会被截断为 3。根因是模型在设计写入接口时选择了"容错"而非"严格校验"，但这违反了对二进制序列化格式的可靠性预期。应改为先校验值的类型与 FieldType 是否匹配，不匹配则抛出 TypeError 或 DeserializationError，避免数据在序列化阶段被无声地篡改。
6. exceptions.py 中定义了 `UnknownFieldError` 异常类（继承自 `SchemaError`），`__init__.py` 也将其纳入了公开导出列表，但在整个 serializer 包的所有源码中从未有任何地方抛出这个异常。`deserialize` 方法遇到未知字段时调用 `_skip_unknown_field` 跳过，跳过失败时抛出的是 `DeserializationError` 而非 `UnknownFieldError`。这说明模型在最初设计异常体系时规划了这个异常类型，但实现时遗忘或改变了处理策略，导致该异常成为孤儿代码。应要么在遇到未知字段且无法跳过时抛出 `UnknownFieldError` 替代通用的 `DeserializationError`，要么从代码中移除这个未使用的异常类。

过程不满意：
1. 模型在收到提示词后的第一个操作就是用 todo_write 创建了 10 项子任务计划，之后才通过 view_folder 和 file_search 查看项目目录结构和 pyproject.toml 等基础设施文件。这种顺序错误导致计划制定时对项目结构、包的命名约定、测试目录布局等缺乏了解，计划的粒度划分只能在后续实施中被动调整。虽然本研究项目中模型的 plan 执行结果尚可，但正确的顺序应该是先读 pyproject.toml 了解项目配置和测试路径设置，再读一个已有的模块（如 rwlock）了解代码风格和目录约定，最后才结合需求制定执行计划。任务规划应该基于对项目环境的充分理解，而非在信息不足时仓促制定。
2. 模型第一次运行测试后发现 1 个用例失败（位于 test_schema.py 的 `test_new_field_id_not_appended_raises`），但没有先阅读完整的测试失败输出和堆栈信息去理解失败根因，而是直接使用 edit_file_search_replace 修改了测试代码。修改前的原测试意图是创建一个新 Schema 其中新增字段的 field_id 小于旧 Schema 最大 id，然后调用 `check_compatibility` 验证抛出错误；修改后的测试变成了创建一个 field_id 重复的 Schema 来触发构造时的 ValueError 校验。这改变了测试的关注点——原来是测试兼容性检查逻辑对新字段 id 必须大于旧 max_id 的约束，修改后变成了测试 Schema 构造时的去重校验，两个层面完全不同。这是典型的试错式修复：看到测试失败不去分析根因就改动测试代码让其通过，没有验证被测代码逻辑的正确性。
3. 模型在执行第二次 pytest 测试时输出了其他包（connpool）的测试结果，说明测试命令的作用域可能超出了 tests/serializer/ 目录。模型随后声称"所有 111 个 serializer 测试全部通过！"，然后还加了一句"后面的 connpool 测试失败是其他模块的问题，与 serializer 无关"。从思考过程的工具调用记录看，模型先后使用了 `cd ... && python -m pytest tests/serializer/ -v`、`python -m pytest tests/serializer -v --tb=short`、`python -m pytest tests/serializer/ -v --tb=short 2>&1 | Select-Object -Last 30` 等不同命令。最后一条使用了 PowerShell 管道只取最后 30 行输出，这种方式可能只看到了测试的尾部汇总信息而错过了关键细节。在未验证完整测试输出的情况下声称 111 个测试全部通过，存在夸大完成度的风险。
4. varint.py 的 `decode_uvarint` 函数（第 29 行）使用 `except Exception as e` 宽泛捕获异常。该函数中唯一可能发生的异常是 `BufferOverflowError`（当 buf 读位置越界时由 Buffer.read_byte 抛出），但此处用 `except Exception` 捕获了所有类型的异常（包括 MemoryError、SystemError 等系统级异常），将其一律包装为 `VarintDecodeError` 再抛出。这种过于宽泛的异常捕获会掩盖编程错误和意料之外的系统错误，让调试变得困难。正确做法是只捕获预期的 `BufferOverflowError` 并转换为 `VarintDecodeError`，其他未预期的异常应直接向上传播。
5. 模型在开发过程中存在多次运行测试使用了不一致的命令格式的问题。第一次用 `python -m pytest tests/serializer/ -v`（带 -v），第二次用 `python -m pytest tests/serializer -v --tb=short`（目录末尾缺斜杠、增加了 --tb=short），第三次用 `python -m pytest tests/serializer/ -v --tb=short 2>&1 | Select-Object -Last 30`（加了 PowerShell 管道截断输出）。这种频繁变更测试命令的行为意味着每次运行的测试范围、输出详细程度和可见输出行数都不一致，无法保证每次看到的结果对应的是同一批测试的全量输出。正确的做法是确定一条标准化测试命令后就始终用它，例如 `poetry run pytest tests/serializer/ -v --tb=short`，确保测试范围、输出格式和完整度一致。
