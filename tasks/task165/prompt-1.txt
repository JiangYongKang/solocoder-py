在当前 Python 项目中实现日志级别动态控制器域，使用内存数据结构管理日志级别配置。
1. 实现按 Logger 粒度的级别覆盖：Logger 按名称标识，采用点号分隔的层级命名规则（如 "app"、"app.service"、"app.service.db"）。每个 Logger 可独立设置日志级别（DEBUG、INFO、WARNING、ERROR、CRITICAL），各级别的数值关系为 DEBUG < INFO < WARNING < ERROR < CRITICAL。允许为任意层级的 Logger 单独设置日志级别而不影响其他 Logger。
2. 实现日志级别的继承传播：当一个 Logger 没有显式设置级别时，应向上查找其父级 Logger 的级别（父 Logger 名称为去掉最后一段点号后的名称）。如果一直未找到显式级别直到根 Logger 也无级别，则使用系统默认级别 INFO。设置父 Logger 的级别后，所有未显式设置级别的子孙 Logger 应立即继承新级别，但已显式设置级别的子孙 Logger 不受影响。
3. 实现运行时热切换：在程序运行过程中可随时调用方法修改任意 Logger 的级别。修改立即生效，之后该 Logger 及受其影响（通过继承）的子孙 Logger 下一条日志即使用新级别。同时提供一个方法检查某条日志（给定 Logger 名称和日志级别）在当前配置下是否应当输出——即日志级别大于等于 Logger 当前生效级别时输出，否则不输出。
4. 代码实现需要放在 src/solocoder_py/log_level/ 包下，然后为该模块编写完整的单元测试，覆盖正常流程（显式设置级别后 is_enabled 判断正确、继承链往上查找正确、修改父级级别后未设级别的子级同步变更、已显式设置的子级不受父级变更影响）、边界条件（根 Logger 级别设置、最深层的继承链、全部级别值的比较正确性）和异常分支（设置无效级别名被拒绝、查询不存在的 Logger 名返回默认级别、同时设置多个 Logger 后继承传播不互相干扰）。测试代码放在 tests/log_level/ 下。
5. 在该模块包内编写 README.md，说明模块功能、核心类的职责、级别继承传播规则与热切换机制（可用文本描述）以及使用示例。
