产物不满意：
1. is_allowed 方法名暗示只是检查，但实际实现调用 try_acquire 成功后没有回滚，每次"检查"都会实实在在消耗配额。提示词第2条要求的是"检查三级配额"，这种有副作用的实现会导致业务方误以为是幂等查询、结果不经意在 is_allowed 就把配额吃掉了。比如配置全局配额为 2，连续调两次 is_allowed("t1", "s1") 都返回 True，但两条记录都写进了全局窗口，之后再调 try_acquire 就直接超限了。
2. try_acquire 中全局→租户→主体三步检查不是原子的。MultiLevelRateLimiter 里的 self._lock 只用于保护 lazy 创建 limiters 的字典操作，没有锁住整个跨层检查流程。三个 SlidingWindowRateLimiter 各自有独立的内部锁，两层锁之间没有协调。两个线程同时 to 同一个租户的请求可能在各层都通过检查后才实际获取，导致整个限流器超出配置配额。
3. 回滚逻辑 _rollback_last 只是简单 pop 最后一个时间戳，不验证这条记录是不是当前请求留下的。并发场景下，线程 A 在全局获取成功后去获取租户配额失败，准备回滚全局，这时如果线程 B 已经获取了全局配额并 append 了自己的时间戳，A 的回滚就会把 B 的记录弹掉，造成 B 的请求被误撤销。
4. RateLimitConfig.validate 没有检查重复的租户 ID。如果配置里有两个同名 tenant_id，MultiLevelRateLimiter.__init__ 构建 _tenant_limiters 字典时后一个会静默覆盖前一个。提示词第3条要求"配置不合法时应拒绝配置"，重复租户 ID 属于不合法但没被拦住。作为对比，subject ID 的重复倒是被检测了，同样的检查没做到租户级别。
5. 测试没有覆盖 is_allowed 的副作用问题。test_is_allowed_returns_false_on_global_limit 连续调了 is_allowed 三次只断言了返回值的 True/True/False，没有检查每次调用后配额计数是否被意外增加。这个测试在掩盖缺陷而不是发现缺陷。
6. 测试缺少并发场景的验证。滑动窗口实现已经用了 threading.Lock，但没有一个测试 case 去验证多线程下的数据一致性和回滚正确性。虽然提示词没有明确要求并发测试，但在代码已经引入了锁的前提下，不写并发测试等于锁是否有效没人知道。
7. README 中描述 is_allowed 为"便捷方法，返回布尔结果"，完全没有提它会消耗配额。文档与代码行为不一致，使用者看了文档会以为这是个安全的查询接口。

过程不满意：
1. 模型在收到任务后第一步就是创建 TodoList（10 个 items），没有先分析提示词的六项需求和三级配额联动模型的技术难点。这不符合标注规范里"不能刚收到任务时就直接创建 TodoList，应先分析问题再规划"的要求。
2. 模型一次性把所有源文件全写出来，写完立马发现回滚逻辑有问题又回去改 sliding_window.py 和 multi_level_limiter.py。说明第一版设计没有在写代码前考虑清楚配额获取失败后如何回滚来保持计数干净，写到一半才发现缺了一块。
3. 测试首次运行出现多个失败后，模型没有逐个分析根因再定点修复，而是一次性改了好几处测试代码（正则匹配、配额总和、未定义租户、滑动窗口过期计数），改完再跑一次看结果。这种"改一堆→跑一遍→不行再改"的模式属于典型的试错式调试，没有体现从报错信息中推理出最小修复点的能力。
4. 运行环境问题折腾了好几次：先 python -m pytest 失败，换 poetry run pytest 也失败，再 pyenv which、pyenv shell 切换 3.13 和 3.11，甚至直接调 Python 3.13 的可执行文件路径也找不到。这几次尝试暴露出模型没有在启动阶段先读 pyproject.toml 确认 Python 版本要求（>=3.13）并检查本地是否装好可用环境，而是边走边试。
5. 模型在最终总结里说"原有 order 模块的 94 个测试全部通过"，但这个信息跟当前任务完全无关。order 模块不是本次任务的产物，评价当前任务不需要拉一个无关模块来撑场面，属于不必要的自我吹嘘。
6. 最终模型用了 pyenv shell 3.11.9 加手动设置 PYTHONPATH 的方式绕开 Poetry 运行测试，而不是去解决 Poetry 本身的 Python 版本不匹配问题。项目用了 Poetry 管理依赖和虚拟环境，绕过构建工具虽然能让测试跑通，但已经偏离了项目约定的运行方式。
