多级滑动窗口限流器有比较严重的原子性和语义问题，需要集中修复一下：
1. is_allowed 方法名为"检查"但实际调用 try_acquire 成功后没有回滚，每次"检查"都会消耗配额。例如全局配额为 2，连续调两次 is_allowed 都返回 True，但两次都写入了窗口记录，之后再调 try_acquire 就直接超限了。is_allowed 不应消耗配额，或者改名并更新文档说明其副作用。
2. try_acquire 中全局→租户→主体的三步检查不是原子的。MultiLevelRateLimiter 的锁只保护字典操作，三个 SlidingWindowRateLimiter 各自有独立内部锁，两层锁之间没有协调。两个线程同时请求同一租户时可能在各层都通过检查后实际配额超出配置上限。
3. 回滚逻辑 _rollback_last 不验证记录归属，直接 pop 最后一个时间戳。并发场景下如果线程 A 在全局获取成功后、租户获取失败准备回滚全局时，线程 B 已经获取了全局配额并写入了自己的时间戳，A 的回滚会把 B 的记录弹掉。
4. RateLimitConfig.validate 检查了重复的主体 ID 但没有检查重复的租户 ID。如果配置中有两个同名 tenant_id，后一个会静默覆盖前一个。补全租户 ID 的重复检查。
5. 测试 test_is_allowed_returns_false_on_global_limit 没有验证每次 is_allowed 调用后配额计数是否被意外修改，掩盖了 is_allowed 的副作用问题。
6. README 中描述 is_allowed 为"便捷方法，返回布尔结果"，没有提它会消耗配额。更新 README 使其与实际行为一致。
7. 修复后重新运行测试，需要让所有测试通过。补充并发场景的测试用例，验证多线程下的数据一致性和回滚正确性。
