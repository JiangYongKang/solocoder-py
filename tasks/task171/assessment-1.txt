产物不满意：
1. canonical.py 的 _symbol_sort_key 函数中，try 块只执行 return (0, symbol)，元组构造从不抛出 TypeError，导致 except 分支是死代码。当频率表包含混合类型符号（如字符串和整数共存）时，sorted 在比较 (0, "A") 与 (0, 1) 阶段会抛出 TypeError，异常发生在 key 函数外部无法被捕获，规范编码生成直接崩溃。根因是错误理解了 Python key 函数的异常传播边界，应改为在排序前对所有符号统一调用 repr 转为稳定可比较字符串。
2. canonical.py 的 build_canonical_codes 方法在处理单一符号时，不管传入的 CodeLengthTable 中该符号的码长是多少，都硬编码 code_length=1 和 code="0"。如果调用者直接传入一个单符号但码长不为 1 的 CodeLengthTable（如 {"A": 5}），生成的 CodeInfo 中码长与实际码长矛盾，导致编码结果与输入码长表不一致。虽然当前通过 extract_code_lengths 进入的路径总是产生码长 1，但作为公开 API 应读取输入码长而非硬编码，根因是单符号分支简化过度，应改为使用 lengths_dict[single_symbol] 替代硬编码值。
3. tree.py 的 extract_code_lengths 中重复实现了 build_huffman_tree 已有的频率表校验逻辑：检查空字典、过滤正频率、验证非空。两处代码各自维护了相同的过滤和校验步骤，如果校验规则需要调整（如增加最小值过滤），两个函数都要同步修改，容易因为遗漏导致两个函数行为不一致。根因是公共校验职责未抽离为独立函数，应将频率表的过滤和校验统一到一个内部函数中。
4. test_edge_cases.py 的 test_reset_decoder 测试用例宣称验证解码器的 reset 功能，但实际在调用 decoder.reset() 之后创建了一个全新的 HuffmanDecoder 对象 decoder2 进行后续操作，reset 后的 decoder 实例从未被再次使用。这意味着 reset 方法的核心逻辑（清空内部缓冲区 _buffer、清空输出列表 _output、重置 _finished 标志）完全没有被测试覆盖到，如果 reset 方法存在不完全清理的缺陷将不会被发现。根因是编写测试时混淆了 reset 复用和新建对象的测试意图，应改为对 reset 后的同一个 decoder 实例执行第二轮写入和 finish 操作。
过程不满意：
1. 模型在刚收到提示词后，仅通过 view_folder 和 file_search 查看了项目目录和文件列表，尚未完整阅读需求文档或分析霍夫曼编码各阶段的依赖关系，就直接用 todo_write 创建了包含 9 项子任务的执行计划（频率统计、树构建、码长提取等）。之后才开始理解需求和参考现有模块代码风格。正确做法是先逐条阅读提示词中的 5 项需求，理解频率统计、码长分配、规范编码生成之间的数据流依赖，再参考现有模块的代码组织方式后制定符合实际模块边界的执行计划。
2. 模型在首次运行 pytest 发现多个测试失败后，采取了反复编辑-运行-失败-再编辑的试错式调试模式。整个调试过程至少经历 5 轮测试运行，每次修改同时涉及 canonical.py、codec.py、models.py 和测试文件等多个文件，而非逐个定位单一根因。test_decode_invalid_code_sequence 测试用例的构造逻辑被三次以上重写（从通用无效序列构造改为手工构造再到改用非满树编码表），却始终没有分析 decoder 是否应该对满二叉树场景也具备无效序列检测能力。正确做法是先阅读失败用例的堆栈信息定位具体出错行，做最小范围的单一修复，通过测试后再处理下一个问题。
3. 模型在调试过程中多次选择修改测试代码来适配生产代码的缺陷，而非根据测试失败信息修复 canonical.py 或 codec.py 中的真正问题。test_edge_cases.py 的增量解码测试被编辑以适配编码器行为，test_error_branches.py 的无效编码测试经历了三轮降级修改——从简化断言逻辑到再次简化再到改用非满码表规避问题。这是典型的测试向生产代码妥协，正确做法是保持测试作为正确性标准不变，根据测试失败信息反向修复生产代码中的缺陷。
