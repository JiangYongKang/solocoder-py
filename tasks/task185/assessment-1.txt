产物不满意：
1. entities.py 顶部定义了三个已编译的正则表达式对象，但 decode_entities 完全没用它们而是用手工状态机逐字符扫描。这导致 import re 成了无效导入、三个正则占内存、维护者看变量名会误以为正则在生效。根因是模型可能最初打算用正则但后来改状态机时忘了删。应该删掉未用的正则和 import re。

2. models.py 基类 Node.__iter__ 用 isinstance(self, Element) 运行时类型判断决定返回子节点列表还是空迭代器，Element 子类没覆写 __iter__。这绕过了多态分发，以后加新节点类型必须同步扩展 isinstance 分支否则新类型迭代退化为空。根因是没遵循模板方法模式在子类各自实现迭代逻辑，应在 Element 子类覆写返回 self.children，Text 子类返回空列表。

3. xpath.py 的 _check_predicates 方法 step 参数类型注解写的是 NameStep，但 _apply_descendant_step 传入的 step 实际是 DescendantStep 实例。虽运行时不影响，但静态类型声明与实际语义不一致用 mypy 会报错。根因是实现两种 Step 后复用 _check_predicates 时没更新类型签名。应改为 Union[NameStep, DescendantStep] 或提取 predicates 属性的基类。

4. parser.py 的 _parse_element 方法直接把 element.children 赋值为子节点列表，完全绕过了 Element 的 add_child 方法。虽然当前因构造时已设父引用所以功能没出错，但如果以后 add_child 加了触发事件维护索引等新逻辑会被静默跳过。根因是解析器内部没统一用 add_child 规范入口。

过程不满意：
1. 模型浏览完项目结构后未读提示词就直接创建了 12 项子任务计划，这时还没理解具体需求。先规划后分析顺序是错的，在理解需求前定详细计划可能跟实际需求错位。虽然本次任务本身线性所以计划还算匹配，但正确做法是先分析提示词的 6 条需求理清模块职责边界和依赖后再按依赖顺序制定计划。

2. 首轮测试发现 20 个失败后模型没读堆栈定位错误行，直接用 Write 重写了整个 parser.py 和 xpath.py。这是大块重写替代定点修复，重写后仍有 13 个失败说明没精准解决之前的失败原因。应该先读每个失败用例的断言和堆栈定位到具体函数行号再做最小范围修改。

3. XPath 求值器经历了至少 7 轮改代码跑测试的循环，其中 // 步进语义和步骤分隔符各经历了方向相反的调整。这种多轮不收敛说明模型没在编码前先设计 XPath 的完整语法和求值语义而是边写边改。应该在纸上先写出 BNF 语法和求值伪代码验证语义正确后再一次性编码。

4. 模型在连续两轮测试修复中使用了 `python -m pytest` 而不是项目规范中的 `poetry run pytest`。虽然功能上在当前环境下等价，但这表明模型没有注意到 pyproject.toml 中定义的 pytest 配置依赖，使用非项目标准的命令可能在某些环境中遗漏项目级别的插件和路径配置。正确做法是使用 `poetry run pytest` 以匹配项目构建工具链。
