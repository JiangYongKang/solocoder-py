产物不满意：
1. 提示词要求统一 TimeoutError 的处理策略解决"定义但不抛出"的割裂，但模型在四个公共方法上增加 raise_timeout 可选参数且默认值 False，保持向后兼容。这意味着默认行为下所有超时路径仍返回 0 或空结果，TimeoutError 依然不抛出，割裂仅在调用方传 True 时才桥接，未改变默认策略。触发场景是调用方用默认参数阻塞读写超时，实际行为是返回 0 或空而非抛异常，业务影响是调用方无法用异常机制统一处理超时。根因是模型优先向后兼容而非让异常成为主通道，应默认抛出 TimeoutError 并提供抑制参数。
2. ring_buffer.py 中 write_batch 和 read_batch 在非阻塞路径完全无视 raise_timeout 参数：write_batch 缓冲区满时 return 0 结束，read_batch 缓冲区空时 return 空列表返回，两处均未检查 raise_timeout。当调用方传 blocking=False 且 raise_timeout=True 时，预期收到异常却静默获得默认返回值，导致误判操作结果。参数名虽暗示只影响超时场景，但非阻塞模式下被静默忽略是 API 设计陷阱。根因是模型未在非阻塞分支加参数校验或文档警告，应在非阻塞且参数为 True 时抛出 ValueError 提示组合无效，或标注该参数仅在阻塞模式生效。
