产物不满意：
1. scripts.py 中 CJK_RANGES 的范围定义存在过度包含问题。提示词要求覆盖 CJK 统一表意文字的基本区 U+4E00 至 U+9FFF 及扩展区 A 至 G，但模型中额外包含了 CJK 符号和标点区 0x3000 至 0x303F、CJK 部首增补 0x2E80 至 0x2EFF、CJK 笔画 0x31C0 至 0x31EF、表意文字描述字符 0x2FF0 至 0x2FFF 等不属于统一表意文字的 Unicode 区块。其中 0x3000 至 0x303F 区间在 CJK_RANGES、PUNCTUATION_RANGES 和 WHITESPACE_RANGES 三个范围列表中同时定义，同一个码点被三个不同的脚本类型声称，虽然 detect_script 的检测顺序保证了实际分类不会出错，但这种重叠定义说明模型在编写范围列表时未对照 Unicode 标准核实每个区间的语义所属，而是把与 CJK 相关但性质不同的所有区块都一揽打进了 CJK 范围。
2. exceptions.py 中定义了 UnsupportedScriptError 异常类，且在 __init__.py 的 __all__ 中公开导出，但全模块没有任何地方实际 raise 该异常。当 tokenizer 遇到 UNKNOWN 脚本的字符时，detect_script 直接返回 ScriptType.UNKNOWN 并由默认规则集处理，并没有抛异常。这个异常类成为死代码，说明模型在编写异常体系时没有设计好异常触发的具体场景，先定义了类名后未落实到调用点上。
3. tokenizer.py 的 UnicodeTokenizer 分词逻辑对英文缩写如 "don't" 的处理存在语义丢失。测试用例 test_contractions 仅断言 "don" 和 "t" 存在于结果中而未验证完整 Token 序列，掩盖了真实行为是生成 ["don", "'", "t", "won", "'", "t", "can", "'", "t"] 的问题，撇号 U+0027 被归入 PUNCTUATION 范围后作为独立 Token 切出，导致 "don't" 被拆成三个 Token 而丢失了缩写语义。虽然提示词未明确要求处理英文缩写，但将标点切成独立 Token 的规则与英文实际使用场景存在冲突，模型应该在 README 中注明这一局限性。
过程不满意：
1. 模型在刚收到提示词后未先阅读 docs 目录下需求文档和项目 pyproject.toml 配置就直接用 todo_write 创建了 12 项子任务计划，之后才去查看现有模块的代码风格。这导致计划的粒度划分与实际模块边界未必匹配，属于未分析先规划的过程偏差。正确做法是先理解需求文档和项目骨架，理清模块职责边界后再制定执行计划。
2. 第一次运行 pytest 后发现测试失败，模型立即判断"一些测试用例的期望值不正确"并开始修改测试文件中的 expected 值，而非先阅读失败用例的堆栈信息定位真实问题。随后经历了至少三轮"改代码 → 改测试 → 再改代码 → 再改测试"的循环，tokenizer.py 和 test_normal_flows.py、test_error_branches.py 在同一轮会话中被反复交叉修改。这是典型的试错式调试，模型遇到失败后不是通过分析错误原因做最小修正，而是同时修改生产和测试两侧代码，来回调整直到凑对为止。正确做法是先读堆栈定位具体错误行与预期行为的差距，只对出错代码做单侧修正，而不是用修改测试期望值来迁就可能有问题的生产代码。
3. 模型在第一轮测试失败后直接假定测试用例期望值有问题，这是一个严重的判断偏差。测试是用例编写者对需求正确性的权威表达，模型应先假定测试期望值是对的、代码逻辑是错的，而不是反过来。尤其对于 Emoji 变体选择符的测试，模型认为"变体选择符被正确跳过了，但测试期望值不对"，于是修改了测试断言来匹配当前的跳转行为，却没有验证跳过变体选择符这个设计决策是否真的符合提示词"代理对字符不破坏切分逻辑"的要求。这属于用修改测试来掩盖代码设计上的不确定决策。
