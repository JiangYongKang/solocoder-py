3. 模型覆盖了 file:///path/to/file、file:///、file://和 mailto: 四种无主机场景，但遗漏了 `http://`（只含 scheme 和双斜杠但无任何主机信息）这类 URL。在 parser.py 的 authority 属性实现中，当 has_authority 为 True 但 host 为 None、userinfo 为 None、port 为 None 时，parts 列表为空导致返回 None，`http://` 恰好触发这条返回路径。该场景未纳入测试意味着这条特定代码路径的回归保护缺位。应补充一个测试验证仅含 `://` 前缀但无主机的 URL 的 authority 为 None。

过程不满意：
1. 模型在收到提示词后的第一步就用 todo_write 创建了 2 项子任务计划，之后才开始用 search_by_regex 搜索 authority 和查看测试文件。这与 assessment-3 已明确指出的"应在先理解代码和需求细节后再制定执行计划"的过程问题重复出现。对于补充回归测试这种需要先了解现有测试覆盖和 authority 语义的任务，两步计划在未读任何文件前制定完全不具备指导价值，且没有包含"分析现有测试覆盖"或"阅读源码"这类关键步骤，导致后续执行中忽略了 test_normal_flows.py 已有的 authority 测试而新增了重复的正例用例。
2. 模型在整个会话过程中从未读取 parser.py 源码文件中 authority 属性和 has_authority 标志的判定逻辑，也从未用 git log 或 git diff 查看 authority 修复的历史变更。回归测试的质量取决于对"回归风险点"的理解——authority 属性有三条独立返回 None 的代码路径：has_authority 为 False 直接返回 None、parts 列表拼接结果为空字符串时返回 None、host 为 None 导致未向 parts 添加内容时返回 None。不了解这三条路径就无法确保测试覆盖了每种触发场景。正确做法是先阅读 parser.py 理解 authority 实现，再用 git 查历史变更确认哪些场景曾被修复过，据此设计最小完备的场景列表。
