产物不满意：
1. 模块存在跨包依赖问题，models.py 中 JWTClock 封装了从 ..seat.clock 导入的 Clock 和 SystemClock 类。seat 包属于另一个任务模块，按项目规范不同任务应有明确的包隔离，jwt 的功能代码不应依赖其他任务包的基础设施。而且 seat 包下 SystemClock.now() 使用 `time.monotonic()` 返回进程相对运行时间而非 UTC 绝对时间，JWT 的 exp、iat、nbf 等时间戳字段在国际标准和工程实践中均使用 UTC 秒级时间戳。在真实部署中如果不显式注入自定义时钟，签发和校验的 exp 计算基准将不是 UTC 时间，导致跨服务 JWT 校验时因时间基准不一致而失败。其他多个包如 websocket 和 credential 均在自身包内独立定义了 Clock 实现，jwt 模块也应该自包含时钟抽象而非外挂依赖。
2. core.py 的 encode 方法在构造 JWT 载荷时无条件用 config.issuer 和 config.audiences 覆盖用户传入 claims 中的 iss 和 aud 字段，代码中 `payload["iss"] = self._config.issuer` 和 `payload["aud"] = list(self._config.audiences)` 直接赋值无任何条件判断。如果用户希望为特定场景签发不同签发者字段的 JWT，例如传入 `{"iss": "third-party-gateway"}` 会被静默替换为 config 中的 issuer 值，没有任何警告或错误提示，用户完全无法感知这种覆盖行为。原因是 encode 没有区分"用户未提供则填入默认值"和"用户已提供则应保留并校验"两种场景，应改为先检查 claims 中是否已有 iss 或 aud 字段，有则校验其值是否合法而非直接覆盖，无则填入配置默认值。
3. _verify_aud 方法在 current_service_id 为 None 时的 fallback 路径 `expected = self._config.audiences[0]` 构成了逻辑自证循环。因为 encode 签发 JWT 时写入的 aud 字段正是 config.audiences 的副本，校验时又从同一个 config.audiences[0] 反推出期望值，只要 JWT 的 aud 数组第一个元素与签发时使用的 audiences[0] 一致，校验就永远通过，current_service_id 的校验收口被彻底架空。测试用例中 test_aud_array_with_multiple_values_hit_first 等虽通过但从未覆盖 current_service_id 为 None 时的场景，这个安全缺陷未被任何测试捕获。正确做法是当 current_service_id 未配置时明确抛出配置错误拒绝校验，而不是退而用签发时写入的值反推期望值。
4. encode 方法中 exp 的填充逻辑为 `if expire_seconds > 0: payload["exp"] = now + expire_seconds`，当 expire_seconds 等于 0 时不会向载荷写入 exp 字段。但 decode 方法的 _verify_exp 要求载荷中必须存在 exp 字段，缺失时抛出 MissingClaimError。JWTConfig 的初始化校验仅拒绝负值而不拒绝零值（`if self.default_expire_seconds < 0: raise ValueError`），这意味着如果用户将 default_expire_seconds 设为 0 或 SignOptions 的 expire_seconds 设为 0，签发的 JWT 永远无法通过自身的 decode 校验。签发端和校验端对 expire_seconds=0 的语义理解不一致：签发端认为 0 表示不设过期时间，校验端却强制要求 exp 必须存在，两端约定自相矛盾。
