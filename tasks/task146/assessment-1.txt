产物不满意：
1. HighlightRegistry 中 set_default_hook 存储了 _default_hook，clear 也会重置，但 highlight 只从 _hooks 字典查找从未回退到 _default_hook。无论调用多少次 set_default_hook 钩子永远不可达。test_default_hook 只验证设置不抛异常未验证钩子被实际使用缺陷未被捕获。根因是只实现了存储和清理出口遗漏核心调用路径的回退逻辑，应在 highlight 中钩子未命中时增加回退分支。
2. _sanitize_attributes 对 href 和 src 做了 javascript: 协议过滤但未处理 data: 协议。攻击者可注入 data:text/html 的 URI，浏览器会将 data URI 中 HTML 作为文档解析执行绕过 XSS 防护。测试只有 javascript 过滤验证无 data URI 测试。根因是只覆盖最常见协议忽略了 data URI 同样是 HTML 注入的重要载体，应增加对 data: 协议的匹配和过滤逻辑。
3. _parse_paragraph 逐行 line.strip() 去掉首尾空白后用 " ".join 合并，但 Markdown 中两个尾随空格加换行应产生 `<br />`。strip() 移除了尾随空格 join 用空格替代换行符，导致 _parse_line_breaks 中替换逻辑永远无法匹配。test_converter.py 完全无硬换行测试。根因是未理解 Markdown 行内空格语义，将预处理和换行符处理拆到解耦错误的两步。应收集段落行时先检测是否以两个空格结尾插入换行标记。
4. _parse_images 和 _parse_links 使用正则 `\(([^)]+)\)` 匹配 URL，遇到 URL 包含括号时提前终止。如 Wikipedia URL `C_(programming_language))` 会解析出缺少右括号的不完整 URL。test_converter.py 无括号链接测试。根因是使用字符排除法匹配括号内容未考虑 URL 自身可含括号。应使用平衡括号计数器或从右向左搜索最后右括号确定 URL 边界。

过程不满意：
1. 模型写完 converter.py 第一版后直接声明有语法问题然后完整重写整个文件，而非先定位具体语法问题所在行做最小范围修正。虽然最终测试通过，但这种发现问题就全量重写的模式效率低下，重写可能引入新缺陷且新旧版本差异难以审查。正确做法是先通过 linter 报错精确锁定错误所在行，针对性修复那一处保持其余已验证逻辑不变。
