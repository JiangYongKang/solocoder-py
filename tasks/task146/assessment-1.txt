产物不满意：
1. 在 `highlighter.py` 的 `HighlightRegistry` 类中，`set_default_hook` 方法存储了 `_default_hook` 属性，`clear` 方法也会重置它，但 `highlight` 方法只从 `self._hooks` 字典查找语言钩子，从未回退到 `_default_hook` 执行默认钩子函数。无论调用多少次 `set_default_hook`，传入的钩子永远不可达。`test_default_hook` 测试用例只验证了设置不抛异常，但未调用 `highlight` 验证钩子被实际使用，该缺陷因此未被捕获。根本原因是模型只实现了默认钩子的存储入口和清理出口，遗漏了核心调用路径中的回退逻辑。正确做法是在 `highlight` 方法中钩子字典未命中时增加 `elif self._default_hook is not None: return self._default_hook(code)` 分支。
2. 在 `sanitizer.py` 的 `_sanitize_attributes` 方法中，对 `href` 和 `src` 属性值做了 `javascript:` 协议过滤，但完全没有处理 `data:` 协议。攻击者可以注入类似 `<img src="data:text/html,<script>alert(1)</script>">` 或 `<img src="data:text/html;base64,...">` 的标签，浏览器会将 data URI 中的 HTML 作为文档解析执行，从而绕过 XSS 防护。test_sanitizer.py 只有 `test_javascript_protocol_with_spaces` 和 `test_javascript_src_removed` 验证 javascript 过滤，没有针对 data URI 攻击的测试。根本原因是模型在构建安全清洗规则时只覆盖了最常见的 `javascript:` 协议而忽略了 data URI 同样是 HTML 注入的重要载体。正确做法是增加对 `data:` 协议的匹配和过滤逻辑。
3. 在 `converter.py` 的 `_parse_paragraph` 方法中，段落行被逐行 `line.strip()` 去掉首尾空白后存入 `para_lines`，最后用 `" ".join(para_lines)` 合并为字符串。但 Markdown 规范定义"两个尾随空格加换行"应产生 `<br />`，例如 `"line 1  \nline 2"` 应输出 `<p>line 1<br />line 2</p>`。`line.strip()` 移除了尾随的两个空格，`join` 用空格替代了换行符，导致 `_parse_line_breaks` 方法中 `text.replace("  \n", "<br />\n")` 的逻辑永远无法匹配——文本中既不存在连续两个空格也不存在换行符。test_converter.py 完全没有硬换行的测试用例。根本原因是模型未充分理解 Markdown 行内空格语义，将预处理和换行符处理拆分到了解耦错误的两个步骤中。正确做法是收集段落行时先检测是否以两个空格结尾并插入换行标记。
4. 在 `converter.py` 的 `_parse_images` 和 `_parse_links` 方法中，使用正则 `\(([^)]+)\)` 匹配 URL 部分。该模式遇到 URL 中包含括号时会提前终止，例如 `[Wikipedia](https://en.wikipedia.org/wiki/C_(programming_language))` 会解析出 URL 为 `https://en.wikipedia.org/wiki/C_(programming_language`，遗漏的右括号残留在输出文本中，生成格式错误的 HTML。维基百科和大量技术文档的 URL 包含括号是真实世界的常见场景。test_converter.py 中没有覆盖包含括号的链接 URL 测试用例。根本原因是模型使用字符排除法 `[^)]` 匹配括号内容但未考虑 URL 自身可包含括号。正确做法是使用平衡括号计数器或从右向左搜索最后一个右括号来确定 URL 边界。
