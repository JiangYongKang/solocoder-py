产物不满意：
1. manager.py 中 _maybe_reset_global_period 在重置所有租户 used 为 0 时只持有 _global_lock 没像 reset_global_quota 那样逐一获取各租户的 tenant_lock。虽当前调用路径都先取得了 _global_lock 保证了安全但两个方法对同一操作使用锁策略不一致，如果将来新增未持锁调用路径将产生并发数据竞争。应在 _maybe_reset_global_period 租户遍历中引入 tenant_lock 获取以保持一致性。
2. models.py 的 GlobalQuota 和 TenantQuota 的 __post_init__ 移除了 used > limit 的校验，模型是考虑到限额调小后 used 可能大于 limit 且 copy() 需正常工作。但移除校验后外部绕过工厂函数直接构造时不会触发错误提示，数据合法性屏障被移除。应保留校验同时修改 copy() 使用 object.__setattr__ 构造副本以绕过 __post_init__ 仅在复制场景允许超限状态存在。
过程不满意：
1. 模型读完三个文件后连续执行了至少七八次代码编辑期间没运行过任何一次 pytest 验证中间状态正确性，直到所有修改完成后才第一次运行测试。这种批量编辑后统一验证的做法增加问题定位难度，多编辑各自引入错误时无法快速判断哪个改动导致哪个失败。应每完成一个独立修复模块就立即运行相关测试子集确保每步验证通过再继续。
2. 模型编写 test_monthly_period_leap_year_feb_29_to_next_feb_non_leap 测试时初始版本时间跳转场景有误：从 2024-02-29 直接 set 到 2025-02-27 跨越十一个月，途中 get_tenant_quota 调用已触发多轮月度重置 used 已变为 0，后续断言失败。说明模型编写时间相关测试时没完整推演时间线中各次 reset 触发顺序和预期结果，应先画出时间线推演每一步 boundary 计算和 reset 触发时机再编码。
