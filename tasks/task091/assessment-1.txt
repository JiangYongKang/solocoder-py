产物不满意：
1. tag_cache.py 的 _purge_expired 方法每次只扫描 _store 字典前 100 个条目且遇到非过期条目就退出，位置 100 之后的过期条目永远不会被清理，导致 size 和 get_stats 返回的计数虚高。根因是扫描策略固定从头部开始且无扫描游标，应记录上次扫描位置或用过期时间排序队列确保覆盖所有过期条目。
2. exceptions.py 中定义的 TagNotFoundError 在导出后从未被抛出，所有标签查询方法对不存在的标签都选择静默返回，与导出异常类的 API 契约矛盾。根因是模型在两个设计阶段决策不一致，应先统一行为——要么删除该异常类，要么在所有查询方法中抛出。
3. tag_cache.py 的 remove_tags 方法没有像 add_tags、invalidate_tag 等其他方法一样校验 `tag is None` 并抛出 InvalidTagError，传入 None 标签时静默失败，而同样的 None 在其他方法中会直接抛异常，API 行为不一致。根因是逐一添加校验时遗漏了该方法，应补齐。
过程不满意：
1. 模型在收到提示词后花费了约 90 步阅读 lru_cache.py、multilevel_cache 等与 tag_cache 无关的模块代码，这些模块属于完全不同的业务域，阅读它们无法为标签倒排索引、悬空清理等任务提供参考，白白消耗了上下文窗口。应直接聚焦于任务指定的目标包和测试目录。
2. 模型在写完 tag_cache.py 约 400 行主体逻辑后没有立即运行 pytest，而是继续写完全部测试和 README 后才一次性运行。如果核心逻辑有缺陷，全写完再测会让问题追溯变困难。应按核心子模块渐进式验证。
3. 模型运行测试时使用 `python -m pytest` 而非项目约定的 `poetry run pytest`，跳过了 poetry 的虚拟环境管理和依赖校验环节，可能因依赖版本差异导致不一致的测试结果。应先阅读 pyproject.toml 了解项目的测试运行方式。
