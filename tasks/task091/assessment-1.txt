产物不满意：
1. tag_cache.py 的 _purge_expired 方法存在过期条目永久残留的缺陷。当缓存中前 100 个条目设置了永不过期（TTL=None），而后续条目有较短 TTL 并已过期时，由于 _purge_expired 每次只扫描 _store 字典的前 100 个条目且遇到非过期条目就退出，位置 100 之后的过期条目永远不会被扫描到。这导致已过期条目长期占用内存，size 属性和 get_stats 方法返回的 entry_count 计数虚高，_expirable_count 也始终保持不准确的值。根因是扫描策略固定从字典头部开始且没有维护扫描游标，应通过记录上次扫描位置或用过期时间排序队列来确保所有过期条目都能被覆盖到。
2. exceptions.py 中定义的 TagNotFoundError 异常是一个从未被抛出的死代码。该异常在 __init__.py 中被导出，但 tag_cache.py 中所有涉及标签查询的方法（get_entries_by_tag 返回空列表，has_tag 返回 False）都选择了静默处理不存在的标签，没有一处抛出 TagNotFoundError。调用方看到导出的异常类会预期不存在的标签将触发异常，但实际行为却是静默返回，造成了 API 契约的不一致。根因是模型在定义异常类时预设了"标签不存在应抛异常"的设计，但在实现核心逻辑时又改用静默策略，两个阶段的设计决策没有保持一致，应删除该异常或统一行为。
3. tag_cache.py 的 remove_tags 方法在遍历标签列表时没有像 set、add_tags、invalidate_tag、invalidate_tags 和 get_entries_by_tag 等其他标签操作一样校验 `tag is None` 并抛出 InvalidTagError。调用方传入包含 None 的标签列表时，remove_tags 会静默调用 entry.remove_tag(None) 返回 False，而同样的 None 标签在 add_tags 中会直接抛出异常。这种 API 行为不一致使得调用方可能因错误参数未被拦截而误以为操作成功，失去了输入校验的防御性保护。根因是模型在为各方法逐一添加 None 校验时遗漏了 remove_tags，应在该方法开头补齐校验逻辑。
