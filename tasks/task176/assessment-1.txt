产物不满意：
1. engine.py 的 _build_bounding_box 方法中，用 lat_offset >= 180.0 来判断是否需要覆盖全球经度，但 lat_offset 是纬度方向的偏移量，用纬度偏移判断经度覆盖范围的逻辑本身是错的。虽然在地球半径 6371 公里、搜索半径约两万公里时碰巧成立，但换成其他地球半径或非地球场景就会暴露。根本原因是混淆了纬度和经度两个独立物理量，应用经度偏移量 lng_offset 来判断。
2. engine.py 的 search 方法中，total_count 被赋值为 len(results) 即截断后的返回结果数，而非实际通过校验的匹配点总数。limit 等于 3 时可能匹配到 10 个点，但 total_count 只返回 3。total_count 的命名让人预期是全部匹配点数，实际等于截断后数量会造成误解。应区分匹配总数和返回结果数两个概念。
3. engine.py 的 search 方法对负 limit 值的校验抛出 InvalidRadiusError 异常，提示信息写的是 Result limit must be >= 0，但类型用的是 Radius 类。limit 是结果数量限制参数，与搜索半径完全无关，用半径异常类描述是一种类型误用。虽然测试也按错误设计来捕获该异常，但根因是异常类型选择时没考虑语义匹配，应为 limit 单独定义或选用通用异常类型。
4. models.py 的 GeoPoint 使用 frozen=True 声明为不可变值对象，但没有实现 __post_init__ 对经纬度做合法性校验。这意味着可以无阻碍构造 latitude=200.0 这样的非法坐标并传入 engine，只有 engine 内部才会拦截。GeoPoint 自身不保证数据完整性，破坏了不可变值对象应维护内部不变量的设计原则。应在 __post_init__ 中加入相同的校验逻辑。
过程不满意：
1. 模型在第一次运行测试后发现 7 个用例失败，随后直接对 engine.py 做了三处不相关的修改——跨反子午线经度包裹处理、add_candidates 原子性、经度偏移计算——且在同一次文件编辑中完成。模型没有先逐一阅读每个失败用例的报错信息和断言内容，也没有逐个修改逐个验证缩小问题范围。不清楚失败原因就一次性批量修改多处属于试错式调试，如果某处引入新错误后续难以追溯根因。
2. 模型在阅读了 ngram 模块代码结构后直接创建了 10 项 TodoList，接着一次性完成 engine.py、models.py、exceptions.py、三个测试文件 86 个用例和 README 的全部编写后，才第一次运行测试。没有按模块粒度逐步验证，例如先写完包围盒就用单元测试验证，再写距离计算用测试验证。全部写完后一次性验证导致 7 个失败需跨模块回溯，降低了调试效率。
3. 模型在 _build_bounding_box 的第二次修复中加入了 lat_offset >= 180.0 的条件，本质是用纬度偏移量判断经度是否需要全局覆盖。lat_offset 是半径除以 KM_PER_DEGREE_LAT 的结果，与经度覆盖范围没有物理因果关系。虽在地球半周长级别大半径下碰巧正确，但反映出模型为让测试通过而堆叠了一个数值上恰好成立但推理上经不起推敲的条件，没有从球面几何原理出发推敲变量选择的正确性。
