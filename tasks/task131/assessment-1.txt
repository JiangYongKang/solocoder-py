产物不满意：
1. processor.py 的 _process_window_closures 方法中，当窗口状态 count 为 0 时空窗口仅被标记 is_closed 但未被从 _windows 字典中删除。158 行的 if 条件限定 count > 0 时才会设置 final_output_emitted，而 163-167 行的清理循环中又仅在 final_output_emitted 为 True 时才 del 窗口，导致空窗口对象被标记关闭后仍滞留在 _windows 中。在长时间运行或窗口粒度较细的场景下，大量空窗口持续累积会造成内存泄漏。根因是空窗口的清理路径被遗漏，应在空窗口过期时直接从 _windows 中无条件删除。
2. processor.py 的 on_event 方法 181-182 行中，当 existing_state 已关闭时，先执行 closure_results = self._process_window_closures() 再抛出 WindowAlreadyClosedError，但返回的 closure_results 变量从未被使用。同样的冗余调用也出现在 186 行的迟到丢弃分支。虽然 _process_window_closures 的副作用（闭窗标记）本身可能需要，但返回值被丢弃且该变量与后续 199 行的 was_fired 和 205 行的 closure_results 变量命名冲突，增加了阅读混淆度。应将这两个异常分支中的 _process_window_closures 调用改为不依赖返回值的纯副作用方法，或直接移除。
3. processor.py 的 _is_event_droppable 方法（104 行）在检查事件是否可丢弃时直接委托给 WatermarkGenerator.is_window_expired 而不判断事件的时间戳是否真的属于该窗口。如果传入的事件时间戳非常小（如 ts=1000 时的 ts=5），_is_event_droppable 仍返回 True 抛出 LateEventDroppedError，但此时异常携带的 window_end 是根据事件时间戳推算出的（如 window_end=1010），而该窗口从未被创建过，异常信息可能误导下游。根因是丢弃判断未与窗口实际存在状态挂钩，应在 _is_event_droppable 中先检查 window_start 是否在 _windows 中再判断过期。
4. processor.py 中定义了 _is_event_late 方法（101-102 行）但在整个模块中未被任何代码调用，属于死代码。该方法的功能已被 _is_event_droppable 完全覆盖，保留未使用的方法增加代码维护负担和读者困惑。应在代码审查时删除该方法或将其重新整合到 on_event 的判断流程中。
5. README.md 中关于迟到事件的三种处理路径的判断流程图（126-140 行）与实际 processor.py 的 on_event 方法（171-206 行）实现逻辑不一致。README 先判断"窗口已关闭"再判断"水位线 >= 窗口结束时间 + 允许迟到时间"，而代码实现中先通过 _is_event_droppable 判断是否丢弃（即路径2），再通过 existing_state.is_closed 判断是否拒绝（路径3），判断顺序与文档描述相反。文档与代码的同步性不足可能导致使用者误解模块行为。
