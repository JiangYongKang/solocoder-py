产物不满意：
1. check_permission 和 require_permission 内部调用 verify_key 导致使用量重复计数。在 manager.py 中，这两个方法直接调用 verify_key，而 verify_key 自身会递增 total_uses 并更新时间戳。若调用方先调用 verify_key 验证密钥再调用 check_permission 检查权限，单次请求使用量被累加两次。原因是 verify_key 将验证和记录耦合在一起，check_permission 把它全量重放了一遍。应拆分为仅校验和校验加记录两个方法，权限检查调用前者，使用量记录由调用方在最外层触发。
2. test_get_effective_scopes_dedup 中存在无效断言。在 test_permissions.py 中，断言 `assert effective.count("c") if isinstance(effective, list) else True` 中 get_effective_scopes 返回 Set 而非 list，isinstance 始终为 False，断言永远通过，去重验证完全无效。原因是测试编写者未核实返回值类型就用 isinstance(list) 条件绕过验证。应直接改用 `assert 'c' in effective` 验证去重。
3. test_multiple_keys_same_subject_independent 中存在无效断言。在 test_edge_cases.py 中，断言 `assert stats1.scopes if hasattr(stats1, 'scopes') else True` 中 UsageStats 没有 scopes 属性，hasattr 返回 False，断言永远通过。原因是编写者不确认 UsageStats 字段结构就写了该断言并用 hasattr 做惰性保护。应删除无效断言或改为验证 UsageStats 实际字段。
4. test_key_prefix_collision_detection 未验证前缀碰撞检测逻辑。在 test_edge_cases.py 该测试中，循环生成密钥检查前缀自然冲突但断言 `assert not found_collision or True` 永远通过，且全程未调用 create_key 触发 manager.py 中的碰撞检测，APIKeyPrefixCollisionError 从未被抛出。原因是靠极低概率随机碰撞验证，还用 or True 使断言无害化。应构造前缀冲突场景再调用 create_key 验证异常抛出。
5. _match_scope_pattern 通配符匹配存在边界歧义。在 models.py 中，`*` 为最后一段时直接返回 True 不检查 scope_parts 是否仍有对应段，导致 `resource:*` 匹配裸 `resource`。resource:* 语义应为命名空间下的子资源而非父级本身。原因是尾缀通配符处理过于宽松，未要求 scope 段数至少等于 pattern 段数。应增加 `len(scope_parts) >= len(pattern_parts)` 约束。
6. check_permission 和 require_permission 存在冗余锁嵌套。manager.py 中这两个方法先用 `with self._lock` 加锁后内部调用 verify_key，verify_key 也加同一把 RLock。虽然 RLock 可重入不会死锁，但嵌套加锁完全冗余，一旦将来 RLock 误改为 Lock 将立即死锁。原因是 verify_key 未区分外部入口和内部调用路径。应拆分为内部无锁版和外部加锁版。
