产物不满意：
1. decompressor.py 的 read 方法存在严重的性能隐患。在每次调用 read 逐点解压时（即 has_more_data 为 True 的情况下循环调用 read），第57行每次都执行 `data = self._input.getvalue()` 来读取完整压缩数据副本，然后仅从中访问 offset 偏移处的几个字节。当压缩数据流较大时（例如压缩后的数据达到 1MB），每解压一个值就需要复制一份完整的 1MB 数据，导致解压过程从 O(n) 退化为 O(n*m)，n 是数据点数，m 是压缩数据总大小。原因是模型在设计 read 方法时没有意识到 getvalue 的复制成本，应改为在 set_input_data 时将原始数据缓存到 self._data 成员变量中，read 方法直接通过 self._data[self._offset] 访问，避免每次读取时重复复制全部数据。

2. varint.py 的 encode_int 函数在第61-62行设置了标记字节的符号位（SIGN_BIT_MASK），指示当前编码值为负数，但 decode_int 函数在解码时完全忽略了这一位——它只在第101-103行校验保留位（bit 0-3 不能为非零），但从不解码或使用 bit 5 的符号标志。实际的符号信息是通过 struct.unpack 用有符号格式（如 `>b`、`>h`）从数值字节中直接解析出来的，标记字节中的符号位纯属冗余。这不仅浪费了格式空间中宝贵的 1 个 bit，还在标记字节的语义上造成了误导——使用者可能认为可以从标记位直接判断符号而跳过实际的数值解析。应删除冗余的 SIGN_BIT_MASK 编码逻辑，或将符号位用于解码时交叉校验（例如 struct.unpack 得到的符号与标记位不一致时抛出数据损坏异常）。

3. README.md 的 varint.py 函数职责表中，decode_int 的返回值描述为"返回（值, 消耗字节数）"，但实际函数签名返回的是三元组（value, consumed, is_anchor），其中 is_anchor 是布尔值指示当前解码的是锚点还是差值。文档遗漏了第三个返回值，可能导致使用者按二元组解包时抛出 `ValueError: too many values to unpack`。原因是模型在后期重构标记字节格式、添加锚点标志位后更新了 decode_int 的返回签名，但遗漏了同步更新 README.md 中对应的 API 文档。应在文档中明确写出三个返回值的含义，例如"返回（整数值, 消耗字节数, 是否为锚点）"。
