你实现的日志段保留与压缩模块存在一些缺陷，需要修复：
1. segment.py 的 append 方法中存在死代码：先检查 `entry.logical_offset == 0 and len(self.entries) == 0` 后将逻辑偏移设为 base_logical_offset，紧接着下一行无条件用 next_logical_offset 覆盖了刚才的赋值。当 base 与 next 不相等时首条记录的偏移就会出错。需要删除冲突的代码行，确保首条记录使用正确的偏移来源。
2. log.py 的 append 方法在压缩期间返回的逻辑偏移不可靠：当正在压缩时，append 将条目入队 pending_writes 后返回当前 next_logical_offset，但 flush 阶段会重新分配偏移，此时值可能已不同。调用者若用 append 返回值去 read，会因偏移不匹配读不到数据。需要保证 append 在压缩期间不返回无效偏移，或压缩结束后自动修正外部持有的偏移引用。
3. log.py 的 __post_init__ 在每次实例化时无条件调用 _create_new_segment 创建空段，导致测试夹具必须用 SegmentedLog.__new__ 绕过构造器来避免段重复。不应让"空的"SegmentedLog 携带初始段，需要改为惰性按需创建首段（首次 append 时才创建）。
4. log.py 的 read 方法中存在冗余的双重偏移解析：先调用 offset_mapper.resolve_logical 获得物理偏移，随后又调用 offset_mapper.get_physical 做第二次解析。两次调用做同一件事，增加了维护复杂度和不一致风险。需要直接复用 resolve_logical 返回的物理偏移。
5. compactor.py 的 compact_segments 方法中，将压缩后条目的新物理偏移赋值给变量 old_physical，变量名与实际语义相反——该变量存储的是压缩后的新偏移值。需要将变量重命名为 new_physical 或 compacted_physical 以反映真实含义。
