产物不满意：
1. segment.py 的 append 方法存在死代码：第27至28行检查 `entry.logical_offset == 0 and len(self.entries) == 0` 后将逻辑偏移设为 `base_logical_offset`，但第29行无条件用 `next_logical_offset` 覆盖了刚才的赋值，两行相互冲突。当 base 与 next 恰好相等时行为正确，但若两者不等（例如从持久化恢复的场景）首条记录就会使用错误的偏移，违背了 base_logical_offset 字段的设计初衷。根因是编写时未注意赋值先后顺序导致的覆盖逻辑。解决方案是删除第27至28行的条件分支或删除第29行让首条使用 base。
2. log.py 的 append 方法在压缩期间返回的逻辑偏移不可靠：当正在压缩时，append 将条目入队 pending_writes 后返回当前 next_logical_offset，但 flush 阶段会重新为条目分配 next_logical_offset，此时值可能已不同于 append 返回时。调用者若用 append 返回值去 read，会因偏移不匹配而读不到数据。根因在于 append 与 flush 各自独立推进 next_logical_offset，缺乏统一偏移分配入口。解决方案是压缩期间不返回有效偏移（返回 -1 或抛异常），迫使调用者在压缩结束后通过其他方式定位写入的条目。
3. log.py 的 __post_init__ 在每次实例化时无条件调用 _create_new_segment 创建空段，导致 make_log_with_old_segments 测试夹具必须用 `SegmentedLog.__new__` 绕过构造器来避免段重复。通过 `__new__` 绕过 __init__ 创建对象是危险的工程做法，暴露了 API 设计缺陷——不应该让一个"空的"SegmentedLog 携带着一个初始段。根因是模型将"需要活跃段写入"的前提固化在构造器中而非惰性按需创建。解决方案是延迟创建首段到首次 append 调用。
4. log.py 的 read 方法存在冗余的双重偏移解析：先调用 offset_mapper.resolve_logical，该方法内部已经通过 global_mapping.get_physical 计算出物理偏移并返回，随后 read 又调用 offset_mapper.get_physical 做第二次解析。两次调用做的是同一件事，无故增加了一次 dict 查找和维护复杂度。如果在两次调用之间映射发生了变更，还会导致读取到的物理偏移不一致。根因是模型未仔细分析 resolve_logical 返回值的完整语义。解决方案是直接复用 resolve_logical 返回的 physical_offset。
5. compactor.py compact_segments 方法中变量命名与语义相反：第83行将压缩后段副本中条目的新物理偏移赋值给变量 `old_physical`，而该变量随后被写入全局和分段偏移映射，实际携带的是压缩后的新偏移值。读者会误以为这里使用了旧的物理偏移做映射更新，而代码实际在做新映射的建立。根本原因是编写时新旧概念混淆导致命名错误。解决方案是将变量重命名为 `new_physical` 或 `compacted_physical` 以反映真实语义。
