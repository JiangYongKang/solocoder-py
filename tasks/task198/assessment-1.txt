产物不满意：
1. bus.py 的 unsubscribe 过滤掉匹配回调后将空列表赋值给通道，但没有在列表为空时从字典中删除该键。README 写明订阅者全移除后通道自动销毁，但自动销毁只在 publish 中存在而 unsubscribe 中缺失。取消全部订阅者后 event_types() 仍返回该类型名称，调用方无法判断通道是否已清空。根因是通道清理逻辑分散在两处且不对称，应在 unsubscribe 末尾补充空通道删除。
2. publish 在锁内复制订阅者列表后释放锁遍历回调，基于复本收集待清理索引，再加锁按索引删除原通道条目。两个线程并发发布同一事件且有待清理条目时，第二个线程基于快照索引删除已被修改过的列表，可能删错正常订阅者。模型加了 RLock 并声称支持并发调用，但没有编写并发测试验证此场景。正确做法是在锁内完成遍历和清理，或将删除改为基于对象身份匹配而非位置索引。
3. subscriber_count 直接返回通道列表的长度，未过滤已失效的弱引用条目。Receiver 对象被回收后但在 publish 触发清理之前，死条目仍残留在列表中，get_callback() 返回 None 但长度未变。此时 subscriber_count 返回包含死条目的膨胀值，与 README 宣称的自动清理不一致，因为清理仅 publish 触发时惰性行为。应遍历过滤或主动触发清理。
4. publish 在锁外执行回调，两个并发调用可能同时获取同一 once 订阅副本。两个线程都独立判断 once 为 True 后都调用回调，导致 once 回调被触发两次，破坏仅一次的语义。虽然通常不会并发发布同一事件，但模型在 README 中主动宣称支持并发调用，而 once 的原子性在并发下未得到保证。应将回调执行与 once 状态清除合并为锁内原子操作。
