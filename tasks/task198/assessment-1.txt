产物不满意：
1. bus.py 的 unsubscribe 方法在列表推导式过滤掉匹配的回调后直接将空列表赋值给 self._channels[event_type]，但没有在列表为空时从字典中删除该键。README 明确写道"当某个通道的所有订阅者都被移除后该通道自动销毁"，然而自动销毁逻辑只存在于 publish 方法中而 unsubscribe 中缺失。后果是取消通道全部订阅者后 event_types() 仍返回该类型名称，调用方无法依据返回值判断通道是否已清空。根因是通道清理逻辑分散在 publish 和 unsubscribe 两处且行为不对称，应在 unsubscribe 末尾补充空通道删除。
2. bus.py 的 publish 方法在锁内复制订阅者列表后释放锁遍历回调，基于复本收集待清理索引，再加锁按索引删除原有通道中的条目。两个线程并发发布同一事件类型且都有待清理的 once 或死弱引用条目时，第二个线程基于快照的索引去删除已被第一个线程修改过的原始列表，可能删除错误的正常订阅者。模型加了 threading.RLock 并在 README 声称"所有操作均支持并发调用"，但没有编写任何并发测试来验证此场景下的正确性。正确做法是在锁内完成遍历和清理，或将删除改为基于对象身份匹配而非位置索引。
3. bus.py 的 subscriber_count 方法直接返回 len(self._channels[event_type])，未过滤 _Subscription 中已失效的弱引用条目。当绑定方法订阅的 Receiver 对象被垃圾回收后但在下一次 publish 触发清理之前，死订阅条目仍残留在通道列表中，其 get_callback() 返回 None 但子列表长度未变。此时 subscriber_count 返回包含死条目的膨胀值，与 README 宣称的"弱引用目标被回收后自动清理"不一致，因为清理是惰性行为仅在 publish 触发时发生。根因是 subscriber_count 没有像 publish 那样对每个条目做活性判断，应遍历过滤或主动触发清理。
4. publish 方法在锁外执行回调，两个并发 publish 调用可能同时获取到同一个 once 订阅的 _Subscription 对象副本。两个线程都独立判断 sub.once 为 True 后都调用 cb(data)，导致 once 回调被触发两次，破坏了提示词第 3 条"执行一次后自动取消订阅"的仅一次语义。虽然调用方通常不会并发发布同一事件类型，但模型在 README 中主动宣称所有操作支持并发调用，而 once 的原子性在并发下并未得到保证。正确做法是将回调执行与 once 状态清除合并为锁内的一个原子操作，或执行前在锁内先检查并标记 once 已触发。
