产物不满意：
1. engine.py 的 deliver 没有检查 next_delivery_at，第一次失败后即使当前时间是 1000 计划重试时间是 1010，外部再次调用 deliver 仍会立即发起第二次投递复现 delivery_count=2，让调用方绕过退避限流故障目标被连续打爆。根因是只在 deliver_all_ready 做到期过滤应在单条投递入口也拒绝未到期消息或统一走调度入口。
2. README 写着 failure_count < max_retries 才重试且 failure_count >= max_retries 转死信，但 RetryStrategy.should_retry 实际允许 failure_count == max_retries 继续重试，max_retries=3 时第 4 次失败才进死信。文档行为冲突会误导使用者配置阈值。根因是修复测试失败后没同步文档语义，应明确最大重试含义并让 README、异常信息和测试保持一致。
3. test_signature.py 的 test_tampered_payload_raises 同时修改了 payload 和密钥，把 payload 从 x=1 改成 x=2 又把密钥改成 wrong_secret，测试失败无法证明是载荷篡改本身被识别。根因是断言变量不单一，应保留正确密钥只修改 payload 并补充对投递 body 与签名计算口径一致性的校验。
过程不满意：
1. 模型第一步就调用 todo_write 创建 8 个待办然后才查看目录，属先规划后理解。这样容易提前固化实现方案，后续虽补看了 retry 和 outbox 模块但没先从提示词拆出重试入口、签名口径、死信语义这些关键约束。应先阅读需求和相邻代码再形成可验证的任务拆分。
2. 模型第一次运行 pytest 出现 4 个失败后一边修改 should_retry 逻辑一边直接调整测试期望值来匹配新语义没有回到提示词确认 max_retries 业务含义，留下 README 与实现不一致问题。应先定义最大重试次数判定边界再同时更新代码、测试和文档。
3. 模型连续两次重读同一文件后只修补括号和误传参数没系统复核签名失败用例是否只改变一个变量，导致 test_tampered_payload_raises 仍用错误密钥掩盖载荷篡改场景。根因是只追求消除当前报错，应在修完显式错误后重新审查同类测试的断言意图。
