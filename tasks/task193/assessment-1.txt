产物不满意：
1. SemverVersion.__init__ 只校验版本数字段，对 prerelease 完全没有格式校验。构造函数接受 prerelease="01"，parse 路径因正则拒绝前置零正确抛异常。根因是 _parse_prerelease_ids 对含前置零的字符串通过 int 静默转换缺格式校验，导致两条 API 对同一输入行为不一致。应在 __init__ 中也对 prerelease 做格式校验或抽取共用校验函数。
2. version.py 的 parse 通过 _SIMPLIFIED_PATTERN 将单数字 "1" 解析为 1.0.0，偏离了 Semver 2.0 规范。规范明确要求版本号必须采用 X.Y.Z 三部分，接受简约形式属于规范偏差。提示词要求按 Semver 2.0 规范解析，简约形式是边界测试而非应接受的输入。根因是模型将 npm 等工具的容错习惯带入应严格遵循规范实现中。应移除回退分支，让单数字统一走 InvalidVersionError 路径。
过程不满意：
1. 模型接到提示词后第一个动作就是 todo_write 创建 5 项计划，然后才用 view_folder 探索项目结构、查看 pyproject.toml 了解约定。这种先计划后分析在尚未理解项目骨架和构建配置时就制定计划，存在粒度与实际边界不匹配的风险。正确做法是先阅读项目目录、构建配置和现有模块风格，理清模块边界后再创建 TodoList。
2. 模型运行 pytest 时用了 python -m pytest 而非 pyproject.toml 中配置的 poetry run pytest。虽然本次环境恰好一致，但非 poetry 的运行方式有使用错误解释器或依赖版本的风险。项目文档多处写明应使用 poetry run pytest，模型忽略了这一约定，可能在依赖不一致环境中产生不可靠的测试结果。正确做法是始终使用 poetry run pytest 确保在虚拟环境中运行。
