产物不满意：
1. service.py 的 submit_csr 在设备已吊销时抛出 DeviceNotRegisteredError。设备实际完成了注册只是被吊销了，异常名却暗示从未注册，与真实状态矛盾。调用方捕获异常后无法从类型上区分"从未注册"和"已被吊销"这两种不同业务场景，影响错误处理的精确性。根因是异常设计时只考虑了未注册而没有考虑已吊销这种合法状态。应新增 DeviceRevokedError 异常，在设备状态为 REVOKED 时抛出。
2. service.py 的 get_device 和 query_certificates_by_device 直接返回 store 内部数据对象的引用。Python dataclass 默认可变，调用方拿到后能直接修改 status 等字段，绕过 service 层篡改 store 内部状态。比如调用方可以不经过 revoke 方法直接把证书 status 从 VALID 改为 REVOKED，导致状态与业务记录不一致。根因是未对返回数据做防御性拷贝，应在返回前深拷贝或创建只读副本。
3. models.py 的 Certificate 定义了 is_expired 方法，但 service.py 的四个核心方法均未调用。过期证书在查询时仍返回 VALID 状态，吊销操作也不因证书自然过期给任何提示。not_before 和 not_after 仅作为存储字段无实际约束力，提示词要求证书包含有效期被弱化为纯存储。根因是 model 层过期判断未与 service 层集成，应在查询证书时统一调用 is_expired，至少让调用方能区分有效期内与自然过期。
过程不满意：
1. 模型收到提示词后第一步就用 todo_write 创建了 7 项计划，随后才阅读 pyproject.toml 和现有代码。在了解项目约定后又重写了 TodoList 从 7 项调为 5 项，说明初始计划未分析就生成，划分粒度与模块职责边界不匹配。这反映出缺少前置分析步骤就制定方案的惯性。正确做法是先阅读需求文档和项目代码结构，理清模块边界后再制定计划。
2. 模型首次运行测试用了 uv.exe run pytest 失败后才改用 python -m pytest。但在此之前已完整阅读过 pyproject.toml，其中明确用 Poetry 管理依赖、脚本都以 poetry 为前缀。模型查看了配置文件却没提取出构建工具和正确测试方式，导致无效调用。这反映处理配置信息时缺乏关联推断能力，正确做法是读 pyproject.toml 后识别构建工具，统一用 poetry run 前缀。
