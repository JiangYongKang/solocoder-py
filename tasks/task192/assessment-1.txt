产物不满意：
1. 在 service.py 的 submit_csr 方法中，当设备状态为已吊销时抛出的是 DeviceNotRegisteredError 异常。但设备实际上已经完成了注册流程，只是被吊销了，异常名称暗示设备从未注册过，与实际业务状态矛盾。这导致 API 调用方在捕获异常后无法从异常类型上区分"设备从未注册"和"设备已被吊销"这两种完全不同的业务场景，影响上层错误处理的精确性和可调试性。根本原因是异常类设计时只考虑了未注册这一种非正常状态，而忽略了已吊销也是一种会导致 CSR 提交失败的合法状态。正确做法是新增一个 DeviceRevokedError 异常，在设备状态为 REVOKED 时恰好抛出该异常，而非复用 DeviceNotRegisteredError。
2. 在 service.py 的 get_device 方法和 query_certificates_by_device 方法中，直接返回了 store 内部存储的 DeviceRecord 和 Certificate 数据类对象的引用。由于 Python 的 dataclass 默认可变，外部调用方拿到返回对象后可以直接修改其 status、device_identifier 等字段，从而绕过 service 层的方法直接篡改 store 内部数据状态。例如调用方可以在不经过 revoke_certificate_by_serial 方法的情况下将某个证书的 status 从 VALID 改为 REVOKED，导致 store 状态与业务操作记录不一致。根本原因是未对返回的内部数据进行防御性拷贝，正确做法是在返回前对可变对象执行深拷贝或使用 dataclasses.replace 创建只读副本。
3. 在 models.py 中 Certificate 数据类定义了 is_expired 方法用于判断证书是否已超过有效期，但 service.py 中的 submit_csr、revoke_certificate_by_serial、query_certificate_by_serial 和 query_certificates_by_device 四个核心业务方法均未调用该方法。已过期的证书在按序列号或按设备 ID 查询时仍然返回 VALID 状态，吊销操作也不会因为证书已自然过期而给出任何提示或语义区分。这意味着证书的 not_before 和 not_after 有效期字段仅作为数据字段保存而无实际业务约束力，提示词要求证书"包含有效期"的语义被弱化为纯存储而非法逻辑校验。根本原因是 model 层的过期判断方法未与 service 层业务逻辑集成，正确做法是在查询证书时统一调用 is_expired 方法，至少让调用方能够区分证书是有效期内还是自然过期。
