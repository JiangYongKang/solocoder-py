产物不满意：
1. manager.py 的 rollback 方法把回滚当成新发布处理生成 v3 之类新版本并设为当前版本，而不是让指定历史版本重新成为当前生效版本。这样 get_current_version 看到新版本号历史列表被额外污染，与提示词中历史版本重新成为当前版本不一致。根因是模型自行加入审计式回滚设计，应按目标历史版本切换当前指针并用该版本触发通知。
2. publish 和 rollback 只在 _compute_changes 返回非空时才调用 _notify_listeners，首次发布空配置、重复发布相同配置、回滚到当前版本这些成功操作都不产生事件，监听方无法得知版本已发布或回滚已执行。根因是把变更键列表非空误当需要通知的前提，应成功发布或回滚都发送事件允许 changes 为空。
3. manager.py 的 get 方法直接返回 ConfigVersion 中保存的值，若值是 dict 或 list 调用方可原地修改并绕过 publish 改写快照，不会生成新版本也不会触发监听通知，破坏版本化发布保存完整配置快照的约束。根因是只给 get_all 和版本查询做深拷贝遗漏单键入口，应对 get 返回值也做深拷贝。
过程不满意：
1. 模型设计回滚机制时没有核对提示词原文，在 README 中写回滚是创建全新版本，随后测试断言 rollback 返回 v3 或 v4 固化了错误语义。根因是过早引入审计式回滚设计，应先按提示词中历史版本重新成为当前版本转成验收断言。
2. 模型编写监听器测试只覆盖新增修改删除等有差异的发布场景，主动写了 test_no_event_when_no_changes 确认不通知，虽 58 passed 但没验证发布成功后监听器应收到版本号这一核心要求。根因是测试围绕实现补强而非需求反例设计，应补充空配置发布和当前版本回滚的通知断言。
3. 模型运行测试后直接总结所有测试通过未继续做需求级反例检查，单键读取泄漏内部可变快照的问题没暴露，热更新数据可能被外部绕过版本系统篡改。根因是把自写单测通过等同于需求满足，应在测试通过后按提示词逐条做行为走查并补齐边界用例。
