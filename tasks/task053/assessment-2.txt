产物不满意：
1. manager.py 的 _compute_changes 方法把内部快照里的值直接塞进 ConfigChange 的 old_value 和 new_value 字段，publish 和 rollback 通知监听器时事件携带的就是内部字典的原始引用。监听器只要原地修改事件里的嵌套字典，比如改写 event.changes 中某条记录的 new_value，内部配置快照就被污染，之后 get 返回的也是被改过的数据，与提示词第 3 条快照隔离的目标相悖。根本原因是模型只修补了 get 这一个读取出口，没有系统性排查所有把可变对象交给外部的路径。正确做法是构造事件时对变更值做深拷贝，或通知前对整个事件深拷贝。
2. manager.py 的 get 方法修复快照泄漏时对返回值无条件执行 copy.deepcopy，连调用方自己传入的 default 参数也会被深拷贝一份再返回。当键不存在时，常见的哨兵对象判断写法 manager.get(key, SENTINEL) is SENTINEL 因为返回的是副本而恒为假，调用方无法区分配置缺失和配置恰好等于默认值，大对象默认值还会带来无意义的拷贝开销。根本原因是模型没有区分取自内部快照的数据和外部传入的默认值这两种来源。正确做法是先判断键是否存在，只对真正取自内部快照的值做深拷贝，键不存在时把 default 原样返回。
3. 回滚改为不创建新版本后，代码里没有任何路径会把 ConfigVersion 的 is_rollback 字段置为真，models.py 里该字段成了永远为假的死字段，而 README.md 的核心类职责表仍写着 ConfigVersion 记录是否为回滚版本，文档与实现不符。调用方按文档通过 get_current_version 返回对象的 is_rollback 判断当前版本是否来自回滚时永远得到假值。模型在改 test_rollback_to_v1 时直接删掉了 rolled.is_rollback is True 断言回避矛盾，却没处理字段语义。正确做法是移除该死字段并同步修订 README 的类职责说明，保持文档和实现一致。
过程不满意：
1. 模型在会话第一步还没读任何代码就直接调用 todo_write 创建任务计划，而且开头连续调用了两次 todo_write，之后才第一次用 view_files 读取 manager.py。在不了解现有回滚和事件触发实现的情况下先定计划，计划只能照抄提示词条目，对修改范围和既有测试的影响面没有任何预判，还多了一次冗余的工具调用，拖慢整体推理效率。根本原因是把建计划当成开场仪式而不是分析结果。正确做法是先读 manager.py 和测试文件，分析三处缺陷涉及的代码路径和受影响的用例，再制定有依据的任务清单。
2. 模型在把 publish 改成无差异也发事件之前已经完整读取过 test_config_hot_reload.py，但首轮只更新了部分用例，漏掉 TestComputeChanges 里两个先发布空配置再取事件列表首元素的测试，第一次运行 poetry run pytest 出现两个失败，之后又重新读取同一个测试文件补上 events.clear()。属于改动影响面扫描不彻底加重复读取同一文件，白白多耗一轮测试和修改。正确做法是改变事件触发语义时先在测试文件里全局搜索所有依赖无变更不发事件假设的断言，一次性同步修改完再跑测试。
