1. resolver.py 的 _resolve_component_by_component 方法中，PathNotFoundError 的 path 字段存在不一致。当逐组件遍历时直接遇到不存在的中间组件（非符号链接且不存在），抛出 PathNotFoundError 携带的是用户传入的原始路径；但当符号链接目标路径内部出现缺失组件时，递归调用产生的异常中 path 字段被设为符号链接的目标路径而非用户原始输入路径。test_resolver_with_nonexistent_symlink_target_intermediate 只验证了 component 字段却未验证 path 字段，未能覆盖这种不一致。调用者收到异常后需要额外追溯符号链接链才能还原原始请求路径，而异常对象本应提供统一且明确的溯源信息。根因是在递归调用处没有将原始入口路径向下传递，应在进入递归时将用户原始路径作为额外参数携带，使得无论在哪一层递归抛出 PathNotFoundError 都能在异常对象中保持原始路径一致。
2. test_error_branches.py 的 test_max_symlink_follows_error_contains_info 测试中使用 `assert "40" in str(exc_info.value)` 来验证异常消息是否包含 max_follows 信息。默认的 max_follows 常量 _MAX_SYMLINK_FOLLOWS 本身就是 40，且测试创建的 50 条符号链接中包含 /link40 的目录路径，导致字符串 "40" 可能来自异常消息模板中的硬编码常量或路径名而并非一定来自 MaxSymlinkFollowsError 构造函数的 max_follows 参数。虽然有 `exc_info.value.max_follows == 40` 这条更强的属性断言做兜底，但字符串断言部分形同虚设未能独立验证异常消息内容。根因是测试用例没有针对异常消息的具体模板格式构造精确的断言，应对照 `Maximum number of symlink follows (40) exceeded` 这一完整子串做匹配而非仅搜索数字 40。
